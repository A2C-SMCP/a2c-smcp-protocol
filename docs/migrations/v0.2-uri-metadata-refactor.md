# v0.2 协议升级指南

**协议版本**：0.1.x → **0.2.0**
**目标读者**：python-sdk / rust-sdk 实现工程师
**发布节奏**：破坏性升级，需同步发布三方（Agent / Computer / MCP Server helper）

!!! tip "Greenfield 阅读提示"

    本指南默认 SDK 已实现 v0.1 完整功能并据此描述"迁移路径"。如果你所维护的 SDK **从未**实现某个子系统（典型如 DPE / Window URI 解析器），那么相关章节应按 **Greenfield 实现规范**阅读：

    - 关于"旧形式"的描述可作为反面参考（避免按 v0.1 模式实现）
    - 关于"新形式"和"行为约束"是 SDK 必须实现的目标
    - "迁移步骤"中提到的"修改 X 文件"对未实现的 SDK 来说是"新建 X 文件"

## 1. 摘要

v0.2 是一次**破坏性**协议升级，包含五组变更：

1. **URI 纯标识符化**：`window://` 与 `dpe://` 都变成纯标识符，不再携带元数据 query；元数据下沉到 MCP `Resource.annotations` + `Resource._meta`
2. **DPE 协议从 Finder 中独立**：v0.1 把"DPE 抽象"和"文档目录浏览器（Finder）"耦合在一起；v0.2 把 Finder 从协议范围**整体移除**，只保留 DPE 文档抽象作为底层原语
3. **DPE 内容投递重构**：DPE 内容**不再走 Socket.IO**——MCP Server `resources/read` 通过双 mimetype（`vnd.a2c.dpe-inline+json` / `vnd.a2c.dpe-uri+json`）自决形态；Computer 端业务层注册 DPE Resolver hook，把 DPE Resource 二次加工为 Agent 可访问的 URI；输入输出 schema 统一为 `A2CResource`
4. **协议版本握手**：引入 Socket.IO connect 阶段的版本校验机制（详见 [versioning.md](../specification/versioning.md)）；4008 显式定义为 HTTP body code，**不是 WS close code**
5. **错误码扩展**：新增 `4006` / `4007` / `4008` / `4011` / `4012` / `4013` / `4014` / `4015`；4013 加 `category` 字段（4 枚举）；4001 描述收紧

### 核心改动一句话版

1. Window/DPE URI 变成**纯标识符**（不再携带元数据 query）
2. 元数据下沉到 MCP `Resource.annotations`（标准字段：`priority` / `audience` / `lastModified`）+ `Resource._meta`（A2C 扩展：`fullscreen` / `keywords` / `file_type` / `page_count`）。**`_meta.file_uri` 已删除**——物理存储位置由 MCP Server 在 `resources/read` 用 `vnd.a2c.dpe-uri+json` 形态显式给出
3. **Finder 整体从协议移除**——`client:list_finder` / `server:update_finder` / `notify:update_finder` / `ListFinderReq` / `DPEDocumentSummary` / `organize_finder` 全部删除；文档发现/检索/聚合视图未来作为内置 MCP Server 重做，与协议核心解耦
4. **新增 `client:get_dpe`**：Agent 把 DPE URI 提交给 Computer；Computer 调 MCP `resources/read` 拿到 inline 文档或 URI 引用，再调本地业务 Resolver hook 二次加工返回最终访问 URI；Resolver 输入输出统一为 `A2CResource`（结构镜像 MCP Resource，A2C snake_case 风格）；协议不规定生命周期/可用性
4'. **新增 `client:get_resources`**：透明转发 MCP 标准 `resources/list` 给 Agent（含 cursor 翻页）；`mcp_server` 必填，Computer 不做协议级过滤；不返回 resourceTemplates（v0.3+ 单独事件）；替代 v0.1 Finder 聚合视图
5. **新增 Socket.IO 连接必带 URL query `?a2c_version=0.2.0`**；Server 在 HTTP 中间件层校验，不兼容时返回 HTTP 400 + `X-A2C-Error-Code: 4008` header + body
6. **MCP Server host 跨 Server MUST 唯一**（注册期硬约束，冲突时注册失败）；host **SHOULD** 使用反向域名风格（lint-style 引导，不符合时 WARN 但不阻塞注册）
7. 元数据字段分工原则：MCP 标准 `annotations`（`priority` / `audience` / `lastModified`）**MUST** 放入 `annotations`；A2C 自定义字段（`fullscreen` / DPE 业务字段）放入 `_meta`。Window `audience=["assistant"]`、DPE `audience=["user"]`

## 2. 受影响范围

### 2.1 代码影响矩阵

> **本表只描述"职能模块"——具体文件路径由各 SDK 自决**。协议指南仅规定每个模块的**职责契约**，不规定 SDK 内部如何组织代码（哪个文件、哪个子包等）。

| 职能模块 | 职责契约 | 说明 |
|---|---|---|
| **WindowURI 解析器** | `window://` URI 解析 + 校验 | 移除 `priority` / `fullscreen` query 解析；含 query 时按分层处理（Agent 构造层失败 / Computer 解析层 WARN 丢弃） |
| **DPE URI 解析器** | `dpe://` URI 解析 + 校验 | URI 纯标识符化（不解析 query）；拒绝 `dpe://host` 形式（无 doc-ref）；分层处理 query / fragment |
| **Desktop 组织职能** | 按 audience / priority 整理 windows，按 size 截断 | 从 `Resource.annotations.priority` 读优先级（float [0,1]）；`Resource._meta.fullscreen` 读全屏标记。具体函数签名 / 参数形式由 SDK 自决 |
| **DPE Resolver Hook（新增）**| Computer 业务层注册接口；把 `(A2CResource, ResolverContents, ResolverHint)` 二次加工为新 `A2CResource` | Resolver 输入输出 schema 见 [data-structures.md](../specification/data-structures.md#dpe-文档相关结构) |
| **MCP Manager** | MCP Server 注册管理；维护 host → server 反查索引 | host 跨 Server 唯一是**注册期硬约束**；冲突时注册失败（不再 WARN + 软约束）|
| **Socket.IO Client（Agent 侧）**| `get_dpe` / `get_resources` 客户端方法 | 删除 `list_finder` |
| **Socket.IO Client（Computer 侧）**| `on_get_dpe` / `on_get_resources` 处理器 | 删除 `on_list_finder` / `update_finder` |
| **DPE mimetype 解析**（新增）| 解析 `vnd.a2c.dpe-inline+json` / `vnd.a2c.dpe-uri+json` 为 `ResolverContents` | 非 vnd 形态 → `4013` + `category=invalid_dpe_mime` |
| **数据类型** | `GetDPEReq` / `GetDPERet` / `GetResourcesReq` / `GetResourcesRet` / `A2CResource` / `ResolverContents` / `ResolverHint` 类型定义 | 删除 `ListFinderReq` / `ListFinderRet` / `DPEDocumentSummary` 等 |
| **错误码枚举** | 新增 `4006` / `4007` / `4008` / `4011` / `4012` / `4013` / `4014` / `4015` | 4013 加 `category` 字段；4001 描述收紧 |
| **协议版本常量** | SDK **MUST** 暴露 `PROTOCOL_VERSION` 常量 | SDK 包版本 MAJOR.MINOR **SHOULD**（非 MUST）对齐协议；详见 [versioning.md SDK 仓库](../specification/versioning.md#sdk-仓库) |
| **Agent/Computer 连接逻辑** | URL query 携带 `a2c_version`；解析 HTTP 400 body 识别 `4008` | 详见 [error-handling.md §4008](../specification/error-handling.md#协议版本不匹配4008) |
| **Server 连接处理** | HTTP 中间件层校验 `a2c_version`，先于 Socket.IO handler | 4008 响应附 `X-A2C-Error-Code` header + body |
| **测试** | URI 解析、Finder 删除、`get_dpe` / Resolver / 4008 / host 唯一性硬约束的测试矩阵 | 详见 [§6 测试矩阵](#6-测试矩阵) |

> **路径标注松绑说明**：v0.2 协议指南只描述职能契约，不再标注具体 Python / Rust 文件路径。各 SDK 自由决定具体放在哪个文件 / 子包——只要外部行为符合契约即可。

### 2.2 非代码影响

- **MCP Server 示例仓库**（若存在）需要更新
- **文档站点**：`docs/specification/`、`docs/guides/`、`docs/appendix/faq.md` 已更新
- **示例 / quickstart**：需要按新模式改写

### 2.3 不变的部分（明确告知，避免过度改动）

- **事件名（Desktop）**：`client:get_desktop` / `server:update_desktop` / `notify:update_desktop` 保持不变
- **Window URI 与 Desktop 行为**：URI 形态、组织算法不变；仅元数据声明位置从 URI query 迁移到 `Resource.annotations` / `_meta`
- **Room 模型 / Server 路由**：完全不变
- **安全模型（零凭证传播、房间隔离）**：完全不变

## 3. 破坏性变更详表

| # | 变更点 | 旧形式 | 新形式 | 兼容策略 |
|---|---|---|---|---|
| B1 | Window URI query 移除 | `window://host/path?priority=80&fullscreen=true` | `window://host/path` | **硬切换**；旧 URI 解析时丢弃 query 并记录 WARN |
| B2 | Window 元数据位置 | URI query | `Resource.annotations.priority`（float [0,1]）+ `Resource._meta.fullscreen` + `Resource.annotations.audience=["assistant"]` | **硬切换**；priority 语义由 int [0,100] 迁移到 MCP 标准 float [0,1]，audience 固定 `["assistant"]` |
| B3 | DPE URI 收缩到 Document 层 | `dpe://host/doc?format=md`、`dpe://host/doc/pages/N`、`dpe://host/doc/elements/ID` | `dpe://host/doc-ref`（doc-ref 可单段或分段路径）| **硬切换**；DPE URI 纯文档标识符化——sub-path（pages/N、elements/ID）与所有 content-control query（`format` / `depth` / `offset` / `limit` / `categories`）全部移除；Page / Element 由 DPE 内容 JSON 在 Document 内部表达 |
| B4 | DPE 文档元数据位置 | Level 0/1 response body 中的扁平字段 | `Resource._meta.{keywords, file_type, page_count}` + `Resource.annotations.lastModified` + `Resource.annotations.audience=["user"]`（注：`_meta.file_uri` 已删除，见 B10）| **硬切换**；audience 固定 `["user"]`（DPE 文档是用户数据资产）|
| **B5** | **Finder 整体从协议移除** | `client:list_finder` / `server:update_finder` / `notify:update_finder` / `ListFinderReq` / `ListFinderRet` / `DPEDocumentSummary` / `DPEPageSummary` / `DPEElementDetail` / `organize_finder` | **不再存在** | **硬删除**；文档发现/检索未来作为内置 MCP Server 重做，与协议核心解耦 |
| **B6** | **新增 `client:get_dpe`** | 无（v0.1 通过 list_finder + MCP `resources/read` 直接返回内容）| `client:get_dpe(uri) → GetDPERet(uri=访问URI)`；DPE 内容**不走 Socket.IO**，由业务 Resolver hook 转成访问 URI（HTTP/file/任意 scheme），Agent 应用层自取 | **新增**；SDK 必须实现该事件；Computer 必须暴露 Resolver 注册 API |
| **B6'** | **新增 `client:get_resources`** | 无 | `client:get_resources(mcp_server, cursor?) → {resources, next_cursor}`；透明转发 MCP 标准 `resources/list`，含 cursor 翻页；不做协议级过滤；为 Agent 提供资源发现入口（替代 v0.1 Finder 聚合视图）| **新增**；Computer 必须实现透明转发；`mcp_server` 必填，不存在返回 404 |
| B7 | host 唯一性 | MUST（v0.1 后期加，跨 Server 强制唯一）| **MUST** 跨 MCP Server 唯一——Computer 在 MCP Server 注册阶段检测，**冲突时注册失败**（硬约束）；不存在运行时 host 路由歧义 | **保持硬约束**；唯一性是协议层硬要求 |
| **B7'** | **host 命名规范**（新增）| 无显式约定 | host **SHOULD** 反向域名风格；SHOULD NOT 单段标识符 / IP / URL 形式；SHOULD 全小写。**lint-style 引导**——不符合时记 WARN 但不阻塞注册 | 详见 [dpe.md host 命名规范](../specification/dpe.md#host-命名规范)。命名风格独立于唯一性硬约束——内网部署用单段标识符在唯一性满足时仍可工作 |
| **B9** | **DPE `resources/read` 双 mimetype**（新增）| MCP Server 自由约定 read 响应形态 | **MUST** 用 `application/vnd.a2c.dpe-inline+json`（小文档 inline）或 `application/vnd.a2c.dpe-uri+json`（大文档 URI 引用）之一 | **硬切换**；非 vnd mimetype → `4013` + `category=invalid_dpe_mime` |
| **B10** | **删除 `_meta.file_uri`** | `_meta.file_uri="file:///..."` 作为原文件 URI hint | **删除**——物理存储位置由 `resources/read` 用 `vnd.a2c.dpe-uri+json` 形态显式给出 | **硬切换**；MCP Server 实现迁移：把 file_uri 改放到 read 阶段返回 |
| **B11** | **Resolver Hook schema 变更**（D3）| `resolve(dpe_uri, contents: list[ResourceContents], hint) → ResolvedResource` | `resolve(resource: A2CResource, contents: ResolverContents, hint: ResolverHint) → A2CResource` | **硬切换**；输入输出统一为 `A2CResource` 形状 |
| **B8** | **Socket.IO 连接握手** | 无版本校验，任意客户端可连接 | **必须**在 URL query 携带 `?a2c_version=...`；Server HTTP 中间件层校验，不兼容返回 HTTP 400 + `4008` | **硬切换**；旧客户端未带 `a2c_version` 被拒（HTTP 400）|
| N1 | MCP 上游授权错误码 | 泛化为 `4003` | `4006` / `4007`（独立码）| 新增；旧 Computer 返回 `4003` 在 Agent 侧仍可工作但体验降级 |
| N2 | DPE 资源访问错误码 | 无 | `4011 DPE Resolver Not Configured` / `4012 Invalid DPE URI` / `4013 DPE Resolution Failed` | 新增；为 `client:get_dpe` 路径提供专属错误语义 |
| N3 | SessionInfo 字段扩展 | `{sid, name, role, office_id}` | 新增 `a2c_version` | 新增字段，对老客户端向后兼容（可忽略不识别字段）|
| **N4** | **新错误码 4014/4015**（E5）| 无 | `4014 MCP Server Not Found` / `4015 MCP Capability Not Supported` | 新增；适用于所有 client:* 事件中 mcp_server 不存在 / 缺 capability 场景 |
| **N5** | **4013 加 `category` 字段**（E1）| `{code: 4013, message, reason}` | `{code: 4013, category: enum, details: object}` | SHOULD 携带；4 枚举：`upstream_unavailable` / `invalid_dpe_mime` / `resolver_error` / `resolver_returned_invalid` |
| **N6** | **4001 描述收紧**（E5）| Tool Not Found：泛指工具不存在 | Tool Not Found：**仅指** `tool_name` 在 server 中不存在；mcp_server 本身缺失用 4014 | 文档澄清；旧 Computer 实现行为可能未变，但描述更精确 |
| **N7** | **4008 显式定义为 HTTP body code**（E3）| 含糊：4008 是错误码 | 4008 **MUST** 在 HTTP 400 body 中传输；**不是** WS close code；HTTP 响应附 `X-A2C-Error-Code: 4008` header 作冗余诊断 | 文档澄清；body 结构加 `min_supported` / `max_supported` 字段 |

## 4. 详细变更规范

### 4.1 Window URI 重构

#### URI 语法

```
// 旧
window://host/path1/path2?priority=P&fullscreen=F

// 新
window://host/path1/path2
```

#### 校验规则

- 解析时若 URI 含 query 部分，**MUST** 丢弃并记录 `WARNING` 级日志
- `path` 段继续支持 URL 编码（`c%2Fd` 保持为单段）
- `host` 跨 MCP Server **MUST** 唯一（注册期硬约束，冲突时注册失败）；命名风格 SHOULD 反向域名（lint-style，不符合时 WARN 但不阻塞）。详见 [§4.4](#44-mcp-server-host-唯一性从-must-软化为-should-又收回为-must-硬约束)

#### 元数据声明（由 MCP Server 在 `resources/list` 响应中设置）

| 字段 | 类型 | 必需 | 默认 | 说明 |
|---|---|---|---|---|
| `Resource.annotations.priority` | `float [0, 1]` | 否 | `0.0` | **MCP 标准**；布局排序依据，值越大越优先，仅同一 Server 内比较 |
| `Resource.annotations.audience` | `["assistant"]` | 推荐 | — | **MCP 标准**；Window 面向 Agent 渲染，MCP Server 应显式声明 `["assistant"]` |
| `Resource.annotations.lastModified` | ISO 8601 | 否 | — | MCP 标准；v0.2 Computer 透传不排序 |
| `Resource._meta.fullscreen` | `bool` | 否 | `false` | **A2C 扩展**；全屏渲染标记 |

#### Desktop 组织职能行为变更

- **职责契约**：从 `Resource.annotations.priority`（float [0, 1]）读优先级；从 `Resource._meta.fullscreen`（bool）读全屏标记；按 priority 在同一 MCP Server 内降序排序；fullscreen per-Server 单独处理（同 Server 内有 fullscreen 时仅保留第一个）
- **不再解析 URI query**——所有元数据来自 Resource `annotations` / `_meta`
- **具体函数签名 / 参数形式由 SDK 自决**——协议指南只规定职能契约，不规定 `organize_desktop(windows, size, history)` 这类具体函数原型；SDK 可选 `(server_name, resource_list, size, history)` 或任何其他形式，外部行为符合契约即可（与 [F5 决议](#84-决议溯源) 一致）

### 4.2 DPE URI 重构

#### URI 语法

```
// 旧（多层 URI + content-control query）
dpe://host/doc-ref?format=markdown&offset=0&limit=20
dpe://host/doc-ref/pages/N?categories=table
dpe://host/doc-ref/elements/ID

// 新（纯文档标识符，只到 Document 一层；doc-ref 可分段路径）
dpe://host/doc-ref
dpe://host/reports/2026/annual                # 分段路径示例
dpe://host/src/main/java/Foo.java             # 含扩展名示例
```

#### 解析器行为

- **拒绝 `dpe://host` 形式**（无 doc-ref）；返回解析失败 / `4012`
- **doc-ref 可分段**：从 host 之后第一个 `/` 起到 URI 结尾的整段 path，整体作为文档唯一标识；**不再有 sub-path（pages/N、elements/ID）概念**——Page / Element 在 DPE 内容 JSON 内部表达
- **拒绝 query / fragment**：解析时检测到则记录 WARN 并丢弃；Agent SDK 构造 URI 时**不应**附加 query 或 fragment
- `host` 跨 MCP Server **MUST** 唯一（注册期硬约束）——`client:get_dpe` 通过 URI 中 host 反查 Server；重复 host 在 MCP Server 注册阶段就会 fail-fast，运行时不存在多匹配场景。详见 [§4.4](#44-mcp-server-host-唯一性从-must-软化为-should-又收回为-must-硬约束)

#### 文档元数据声明（替代旧的 URI query / Level 0 body）

每个文档作为一个 `Resource` 出现在 `resources/list` 响应中：

```python
Resource(
    uri="dpe://com.example.docs/rpt-2026",
    name="2026 年度报告",                              # = title
    description="2026 年度财务与运营报告",              # = summary
    mimeType="application/json",
    annotations=Annotations(
        audience=["user"],                            # DPE 文档是用户数据资产，固定 ["user"]
        lastModified="2026-01-15T08:30:00Z",
    ),
    _meta={
        "keywords": ["财务", "年报"],
        "file_type": "xlsx",
        "page_count": 12,                             # 推荐：Agent 据此规划翻页
    },
)
```

> **`_meta.file_uri` 已自 v0.2 删除**——物理存储位置由 MCP Server 在 `resources/read` 阶段用 `application/vnd.a2c.dpe-uri+json` 形态显式给出，避免 list 阶段 hint 与 read 阶段事实来源冲突。

#### 不再有 v0.1 的 Level 0/聚合端点

`dpe://host` 形式（v0.1 用作 catalog 端点）整体移除。文档发现走 MCP 标准 `resources/list` 或未来的 Finder MCP Server。

详见 [DPE 文档协议](../specification/dpe.md)。

### 4.3 client:get_dpe 与 DPE Resolver Hook（新增）

#### 设计动机

v0.1 的 DPE 内容通过 MCP `resources/read` 直接经 Socket.IO 返回给 Agent。**对大体量文档（Excel / PDF / PPT 数十 MB）这是不可行的**——Socket.IO 默认 `maxHttpBufferSize` 1MB；即便调大，单消息阻塞通道、Server 全量缓冲等问题都会暴露。

v0.2 把 DPE 内容投递与 A2C 传输层解耦：

```
Agent → client:get_dpe(uri) → Computer → MCP resources/read（含双 mimetype）
                                                │
                                                ▼
                                       DPE Resolver Hook → 访问 URI
                                                │
                                                ▼
                                       业务存储（OSS / 本地 / etc）
                                                │
        Agent ← 应用层协议（HTTP / file / ...）拉取 ←
```

A2C 协议本身**不**承载 DPE 内容；URI 的生命周期、可用性、签名机制全部交给业务方。

#### `resources/read` 双 mimetype 形态（D2）

DPE 文档的 `resources/read` 响应 **MUST** 使用以下两种 A2C 协议私有 mimetype 之一：

| mimetype | 形态 | 适用 |
|----------|------|------|
| `application/vnd.a2c.dpe-inline+json` | text 字段为 DPE 标准 JSON | 小文档（< 1MB 量级）|
| `application/vnd.a2c.dpe-uri+json` | text 字段为 URI 引用 JSON `{uri, mime_type, size, expires_at?, etag?}` | 大文档 / 已在 OSS 等外部存储 |

MCP Server **SHOULD** 基于 `Resource.size` 阈值（典型 1MB）自决形态。Computer **MUST** 拒绝任何**非** vnd 形态 → `4013` + `category=invalid_dpe_mime`。

详见 [dpe.md DPE 文档读取的两种形态](../specification/dpe.md#dpe-文档读取的两种形态-mimetype)。

#### 新增事件

| 事件常量 | 事件名称 | 数据结构 |
|---|---|---|
| `GET_DPE_EVENT` | `client:get_dpe` | `GetDPEReq` → `GetDPERet` |

```python
class GetDPEReq(AgentCallData, total=True):
    agent: str
    req_id: str
    computer: str
    uri: str                  # dpe://host/doc-ref（doc-ref 可单段或分段路径）
    timeout: NotRequired[int]


# GetDPERet = A2CResource + req_id（A2CResource 字段集见 data-structures.md）
class GetDPERet(A2CResource, total=False):
    req_id: str               # A2C 协议级关联字段
```

#### Resolver IO schema（D3）

Resolver **输入输出统一为 `A2CResource`**——结构镜像 MCP Resource，A2C snake_case 风格。Resolver 是 MCP Resource 的二次加工器：把"Agent 想读的 DPE URI"映射到"Agent 可以访问的 URI"。

```python
# Resolver 签名（语言无关契约）
resolve(
  resource: A2CResource,           # 原始 DPE Resource，uri = dpe://...
  contents: ResolverContents,       # 解析后的内容形态（Inline | External）
  hint: ResolverHint,               # 协议级运行时上下文
) -> A2CResource                    # 新 Resource，uri = Agent 可访问 URI
```

完整 schema（A2CResource / ResolverContents / ResolverHint）见 [data-structures.md DPE 文档相关结构](../specification/data-structures.md#dpe-文档相关结构)。

#### Computer 必须实现 DPE Resolver 注册 API

```python
# Python reference impl
class DPEResolver(Protocol):
    async def resolve(
        self,
        resource: A2CResource,
        contents: ResolverContents,
        hint: ResolverHint,
    ) -> A2CResource: ...


# Computer SDK 启动时业务方注册
computer.register_dpe_resolver(MyOSSResolver())
```

未注册时收到 `client:get_dpe` 必须返回 `4011 DPE Resolver Not Configured`——**禁止**降级到 inline 透传 ResourceContents。

#### Computer 处理流程

1. 接收 `client:get_dpe(uri=dpe://...)`
2. 校验 URI（dpe scheme + 非空 host + 非空 doc-ref）；非法返回 `4012`；含 query / fragment 时按 [分层处理](../specification/dpe.md#uri-query--fragment-处理分层) 容错丢弃 + WARN
3. 检查 Resolver 是否注册；未注册返回 `4011`
4. 解析 host → 定位对应 MCP Server；找不到 → `4014 MCP Server Not Found`；server 未声明 `resources` capability → `4015 MCP Capability Not Supported`
5. 调 MCP Server 的 `resources/read(uri)` 拿 `ReadResourceResult`
6. 解析 mimetype → 构造 `ResolverContents`：
    - `application/vnd.a2c.dpe-inline+json` → `Inline { document }`
    - `application/vnd.a2c.dpe-uri+json` → `External { uri, mime_type?, size? }`
    - 其他 → `4013` + `category=invalid_dpe_mime`
7. 调 Resolver `resolve(resource, contents, hint)` → 新 `A2CResource`；Resolver 抛异常 → `4013` + `category=resolver_error`；返回值非法 → `4013` + `category=resolver_returned_invalid`
8. 包装为 `GetDPERet`（追加 `req_id`），返回给 Agent

详见 [DPE 文档协议](../specification/dpe.md)。

### 4.4 MCP Server host 唯一性（从 MUST 软化为 SHOULD 又收回为 MUST 硬约束）

v0.2 决议：host 跨 MCP Server **MUST 唯一**（注册期硬约束）。早期讨论曾考虑过 SHOULD + WARN 软约束，最终确认：软约束在生产环境下让 host 重复故障不可观测、不可恢复，运维代价远高于注册期硬失败。

#### 路径与路由依据

| 协议路径 | 路由依据 | host 角色 |
|---|---|---|
| `client:get_dpe(uri=dpe://host/...)` | URI 中的 host 反查 MCP Server 索引 | **路由关键字**——保持 URI 自包含寻址能力 |
| `client:get_resources(mcp_server=name)` | `mcp_server` 字段直接定位 | 不参与路由 |
| `client:get_desktop` | desktop 组织职能按 MCP Server 名称分组 | 仅参与单 Server 内的窗口排序 |

#### Computer 端实现要求（v0.2 硬约束）

| 阶段 | 行为 |
|---|---|
| **MCP Server 注册时** | Computer **MUST** 检测新 Server 的 host 是否与已注册 Server 冲突；**冲突时注册失败**（硬约束），返回明确错误（含两个 Server 名称 + 共用 host）|
| **`client:get_dpe` 路由时** | (a) host 唯一匹配 → 路由到该 Server；(b) 无匹配 → 返回 `4014 MCP Server Not Found`；**不存在多匹配场景** |

#### 设计取向

- **URI 自包含寻址**：Agent 持有任意 dpe URI 即可调 `get_dpe`，不需要外部元信息
- **注册期硬约束**：host 重复是 MCP Server 命名 bug，**MUST 在启动期发现并修复**；Computer 不允许带病运行
- **运行时无歧义**：注册期硬约束保证了路由确定性——同一 URI 在任意时刻、任意部署上路由结果稳定

#### host 命名规范（B7'）

host **SHOULD** 反向域名风格：`<tld>.<organization>.<service>[.<sub>]*`。详见 [dpe.md host 命名规范](../specification/dpe.md#host-命名规范)。

> **强度说明**：命名规范是 **lint-style 引导**（SHOULD），不是注册期硬约束。唯一性才是硬约束（B7 MUST）。不符合规范的 host 在注册时记 WARN 但仍允许注册成功——为内网部署 / 简化命名场景留出灵活性。

```
✅ 推荐：com.example.docs / com.example.code / com.example.mail
⚠️  允许但 WARN：docs / DOCS.EXAMPLE.COM / docs.example.com:8080 / docs_server
```

#### 运维兼容性检查清单（v0.1 → v0.2 升级前 MUST 验证）

启用 v0.2 硬约束前，运维 / 业务方需逐项核对：

- [ ] 列出当前 Computer 上所有 MCP Server 的 host 声明
- [ ] 检查是否存在跨 Server 的 host 重复（**唯一性是 v0.2 硬约束，重复会导致注册失败**）
- [ ] 检查是否有 host 不符合反向域名规范（单段标识符 / IP / URL 形式）——这是 SHOULD 级 lint 提示，不阻塞注册但建议修复
- [ ] 修复发现的命名冲突（推荐采用 `<tld>.<organization>.<service>` 风格统一规划）
- [ ] 升级到 v0.2 后通过 fail-fast 启动验证（host 冲突会让 Computer 启动失败而非静默运行；命名风格不规范仅产生 WARN 日志）

升级 v0.2 时若 Computer 启动失败报"host 重复"，**MUST**先修复 MCP Server 命名再启动——不要试图用 v0.1 SDK 旁路。

详见 [DPE host 路由策略](../specification/dpe.md#host-路由策略) / [Window URI 校验规则](../specification/desktop.md#window-uri-规范)。

### 4.5 错误码新增

#### MCP 上游授权（4006 / 4007）

| 代码 | 名称 | 含义 |
|---|---|---|
| `4006` | Tool Authorization Required | 工具需要 MCP 上游授权（如 OAuth），Computer 当前无有效凭证 |
| `4007` | Tool Authorization Failed | 上游授权流程失败、Token 失效、刷新失败、权限不足 |

详见 [error-handling.md 4006/4007 判定决策表](../specification/error-handling.md#40064007-判定决策表)。

#### DPE 资源访问（4011 / 4012 / 4013）

| 代码 | 名称 | 含义 |
|---|---|---|
| `4011` | DPE Resolver Not Configured | Computer 未注册 DPE Resolver hook |
| `4012` | Invalid DPE URI | Agent SDK 构造层硬错误——URI 不符合 `dpe://` scheme 规范 |
| `4013` | DPE Resolution Failed | Resolver / 上游 / mimetype 失败——payload **加 `category` 字段**细分 |

#### 4013 `category` 字段（v0.2 新增，SHOULD 携带）

| `category` | 触发场景 | Agent UX 建议 |
|------------|----------|---------------|
| `upstream_unavailable` | MCP Server 不可用 / `resources/read` 超时 / 上游 5xx | 提示"DPE 来源暂时不可用"，**可重试** |
| `invalid_dpe_mime` | MCP Server 返回非 `application/vnd.a2c.dpe-inline+json` / `application/vnd.a2c.dpe-uri+json` 的 mimetype | 提示"DPE 来源实现不规范"，**不要重试** |
| `resolver_error` | Resolver Hook 抛异常 | 提示业务层故障，**不要重试** |
| `resolver_returned_invalid` | Resolver 返回的 `A2CResource` 缺 `uri` / 字段非法 | 提示业务层 bug，**不要重试** |

枚举值 v0.2 锁定 4 个；新增 category 不算 breaking（minor 升级即可）；删除 / 重命名算 breaking；不允许业务自定义 category。

#### MCP Server 路由（4014 / 4015，v0.2 新增）

| 代码 | 名称 | 含义 |
|---|---|---|
| `4014` | MCP Server Not Found | 引用的 `mcp_server` 名字未注册（适用于所有 client:* 事件）|
| `4015` | MCP Capability Not Supported | MCP Server 已注册但未声明所需 capability（如 `client:get_resources` 时 server 缺 `resources` capability）|

#### 4001 描述收紧

| 旧描述 | 新描述 |
|--------|--------|
| Tool Not Found：工具不存在 | Tool Not Found：**仅指** `tool_name` 在指定 MCP Server 中不存在；mcp_server 本身缺失用 4014 |

详见 [error-handling.md DPE 资源访问错误](../specification/error-handling.md#dpe-资源访问错误)。

### 4.6 协议版本握手（新增）

#### 为什么选 URL query

A2C 协议把 `a2c_version` 放在 **URL query**，不是 `auth` dict 或自定义 HTTP header。三方案权衡：

| 方案 | HTTP 层可见 | SDK 透传一致性 | 反代拦截 |
|------|-----------|---------------|---------|
| **URL query**（采纳） | ✅ | ✅ 所有 socketio 客户端原生支持 | ✅ |
| `auth` dict | ❌ 在 Socket.IO 业务层之后才可见 | ✅ | ⚠️ |
| `X-A2C-Protocol-Version` header | ✅ | ⚠️ 各 SDK 透传 header API 一致性较弱 | ✅ |

核心理由：版本校验**必须先于 Socket.IO handler 完成**——`auth` dict 不满足；URL query 是 socketio 客户端最一致的 HTTP 层透传方式；`a2c_version` 不是敏感信息，access log 记录无安全风险。详见 [versioning.md §为什么选 URL query](../specification/versioning.md#为什么选-url-query)。

#### 客户端（Python reference impl）

```python
import socketio
from a2c_smcp import PROTOCOL_VERSION

await sio.connect(
    f"wss://server.example.com?a2c_version={PROTOCOL_VERSION}",
    socketio_path="/smcp",
    auth={"role": "agent"},
    transports=["polling", "websocket"],   # SHOULD：保证首个握手是 HTTP polling，4008 body 可访问
)
```

> 协议规范仅提供 Python reference impl。其他 SDK（Rust / TypeScript / Go 等）的连接示例由各 SDK 自行决定——A2C-SMCP 协议文档不堆砌多语言示例。

#### Server

HTTP 中间件层校验 `a2c_version`，不兼容返回 HTTP 400 + `X-A2C-Error-Code: 4008` header + body。详细规范见 [versioning.md](../specification/versioning.md) 与 [error-handling.md §4008](../specification/error-handling.md#协议版本不匹配4008)。

#### 4008 显式定义为 HTTP body code

!!! warning "明确语义边界"

    **4008 是 ErrorPayload.code 字段值，承载于 HTTP 400 响应 body 中**。
    4008 **不是** WebSocket close code（RFC 6455 自定义码段 4xxx）——协议版本校验在 HTTP 层完成，发生在 WS 帧建立之前，不存在 WS close 形态。
    SDK 实现 **MUST NOT** 把 4008 与 WS close code 4xxx 混淆。

#### HTTP 响应 body 结构

```json
{
  "code": 4008,
  "message": "Protocol version mismatch",
  "server_version": "0.2.0",
  "client_version": "0.1.5",
  "min_supported": "0.2.0",
  "max_supported": "0.2.999"
}
```

`X-A2C-Error-Code` 响应 header 是冗余诊断辅助：当客户端无法访问 body 时（罕见 transport 边角情况），可从 header 中识别错误类型。**不替代** body，body 是 single source of truth。

#### Client SDK 收到 4008 后必须主动 disconnect

防止底层库自动重连导致死循环。详见 [versioning.md § 4. Client SDK MUST 主动断开连接](../specification/versioning.md#4-client-sdk-must-主动断开连接)。

## 5. 迁移步骤

### 5.1 MCP Server 实现者

1. **Window**：把 URI query 中的 `priority` / `fullscreen` 改成 Resource 元数据
   ```python
   # 旧
   Resource(uri="window://host/path?priority=80&fullscreen=true", ...)

   # 新
   Resource(
       uri="window://host/path",
       annotations=Annotations(priority=0.8, audience=["assistant"]),  # priority int [0,100] → float [0,1]
       _meta={"fullscreen": True},
   )
   ```

2. **DPE Resource 元数据**：把所有 query 参数移除；把文档元数据迁到 Resource 上（**`_meta.file_uri` 已删除**）
   ```python
   Resource(
       uri="dpe://com.example.docs/doc-ref",                           # host SHOULD 反向域名风格（不符合时 WARN）
       name="...", description="...", mimeType="application/json",
       annotations=Annotations(audience=["user"], lastModified="..."),
       _meta={"keywords": [...], "file_type": "xlsx", "page_count": 12},  # 不再有 file_uri
   )
   ```

3. **DPE `resources/read` 响应**：MUST 用 A2C 私有 mimetype 之一
   ```python
   # 小文档 inline 形态
   TextResourceContents(
       uri="dpe://com.example.docs/doc-ref",
       mimeType="application/vnd.a2c.dpe-inline+json",
       text=json.dumps({"doc_ref": ..., "pages": [...]}),  # DPE 标准 JSON
   )

   # 大文档 URI 引用形态（避免 base64 编码瓶颈）
   TextResourceContents(
       uri="dpe://com.example.docs/doc-ref",
       mimeType="application/vnd.a2c.dpe-uri+json",
       text=json.dumps({
           "uri": "https://oss.example.com/.../doc.pdf",
           "mime_type": "application/pdf",
           "size": 12_345_678,
           "expires_at": "2026-04-27T12:00:00Z",
       }),
   )
   ```

4. **host 命名规范**：SHOULD 反向域名风格（`com.example.<service>`）；不符合时协议层 WARN 但不阻塞注册（lint-style 引导）。**唯一性才是硬约束**（注册期检测，冲突即失败）

### 5.2 Computer SDK 工程师

1. **WindowURI 解析器**：移除 query 解析；解析时若有 query，丢弃并记录 WARN（Computer 解析层容错）；Agent SDK 构造层则 MUST 校验失败
2. **DPE URI 解析器**：拒绝 `dpe://host` 形式（无 doc-ref）；query / fragment 按分层处理
3. **Desktop 组织职能**：从 `Resource.annotations.priority`（float [0,1]）读优先级；从 `Resource._meta.fullscreen` 读全屏标记。具体函数签名 / 参数形式由 SDK 自决
4. **删除 organize_finder / Finder 事件处理**：v0.2 协议无 Finder
5. **新增 DPE Resolver 注册 API**：暴露注册接口；Resolver 签名 `resolve(resource, contents, hint) -> A2CResource`
6. **新增 `client:get_dpe` 处理器**：按 [§4.3 Computer 处理流程](#computer-处理流程) 实现
7. **新增 mimetype 解析**：`vnd.a2c.dpe-inline+json` → `Inline` / `vnd.a2c.dpe-uri+json` → `External` / 其他 → `4013` + `category=invalid_dpe_mime`
8. **MCP Manager**：(a) MCP Server 名称索引（用于 `client:get_resources` / `client:tool_call` 直接定位）；(b) host → MCP Server 反查索引（用于 `client:get_dpe`）；**注册期检测 host 重复 → 注册失败**（硬约束）
9. **错误码枚举**：新增 4006 / 4007 / 4008 / 4011 / 4012 / 4013（含 category）/ 4014 / 4015；4001 描述收紧

### 5.3 Agent SDK 工程师

1. **删除 `list_finder` 方法**
2. **新增 `get_dpe(uri)` 方法**：返回 `GetDPERet`（= `A2CResource + req_id`）；Agent 业务代码拿到访问 URI 后用应用层协议自取实际内容
3. **新增 `get_resources(mcp_server, cursor=None)` 方法**：透明转发 MCP `resources/list`；返回 `{resources, next_cursor}`；`mcp_server` 必填
4. **HTTP / file URI 内置拉取**：建议 SDK 内置 http(s):// 与 file:// 的拉取能力；其他 scheme 由用户业务代码处理
5. **不缓存 URI 跨 session**：拿到 URI 仅在本次会话使用；过期时重新调用 `get_dpe`（`_meta.expires_at` 给出过期时间）
6. **协议版本握手**：连接 URL query 拼 `?a2c_version=...`；解析 HTTP 400 body 识别 4008 → 主动 `disconnect()` → 抛 `ProtocolVersionError`
7. **DPE 发现链路**：典型路径是 `get_config(computer) → 对每个 server name 翻页 get_resources → 按 scheme/元数据自决过滤 → get_dpe`
8. **URI 构造层校验**：构造 dpe URI 时检测到 query / fragment **MUST** 校验失败抛异常（与 Computer 解析层容错丢弃形成分层）

### 6.2 organize_desktop（与 v0.1 保持算法一致，仅元数据来源变化）

## 6.8 priority int [0,100] → float [0,1] 迁移（F2）

WindowURI.priority 在 v0.1 是 `int [0, 100]`，v0.2 改为 `float [0, 1]`（值域 + 类型双变）。MCP Server 实现侧 **MUST** 做换算迁移——直接传 v0.1 风格的 `int` 值会被 v0.2 校验失败（`80 > 1` 越界）。

#### 换算对照表

| v0.1 priority (int) | v0.2 priority (float) | 语义 |
|---------------------|------------------------|------|
| 0 | 0.0 | 最低（默认）|
| 25 | 0.25 | 较低 |
| 50 | 0.5 | 中等 |
| 80 | 0.8 | 较高 |
| 100 | 1.0 | 最高 |

**换算公式**：`priority_v0_2 = priority_v0_1 / 100.0`（整数除以 100，结果保留 1-2 位小数）

#### 迁移示例

```python
# v0.1 旧形式
Resource(uri="window://host/path?priority=80&fullscreen=true", ...)

# v0.2 新形式（priority 80 → 0.8）
Resource(
    uri="window://com.example.editor/path",
    annotations=Annotations(priority=0.8, audience=["assistant"]),
    _meta={"fullscreen": True},
)
```

MCP Server 升级时遗漏换算的常见症状：v0.2 Computer 启动后 Desktop 列表为空 / priority 校验告警刷屏——检查日志中 `priority=80, expected float in [0, 1]` 这类信息即可定位。

## 7. 兼容性策略

### 7.1 协议层：硬切换

v0.1.x 与 v0.2.0 在 URI 形态、Finder 移除、`get_dpe` 新增等多处**无法互通**。Server 端通过 HTTP 中间件 `a2c_version` 校验，不兼容直接拒绝（HTTP 400 + `4008`）。

#### 不内建 v0.1 MCP Server fallback

v0.2 SDK **MUST NOT** 内建针对 v0.1 MCP Server 的 URI query fallback 解析。当 v0.2 Computer 遇到只在 URI query 中携带 `?priority=` / `?fullscreen=` 的 v0.1 MCP Server 时：

- 按 v0.2 校验规则丢弃 URI query 部分（仅 WARN）
- 不读取 `_meta` / `annotations` 之外的元数据来源
- Desktop 中该窗口降级显示为 `priority=0.0, fullscreen=false`

理由：保持协议纯粹；用户感知到降级会主动升级 MCP Server，比静默兼容更早暴露问题。

### 7.2 数据层：Agent 视角

- Window 端 Agent 看到的 `Desktop` 字符串列表形态**不变**
- DPE 端 Agent 拿到的是访问 URI（v0.1 没有这个语义，是新增）；具体内容由应用层拉取

### 7.3 MCP Server 生态

- 旧 MCP Server（只输出 URI query 形态）在 v0.2 Computer 下 Desktop **MUST 降级**为 `priority=0.0, fullscreen=false` 并 WARN
- 旧 MCP Server（实现 v0.1 DPE Level 0）在 v0.2 Computer 下，Level 0 响应被忽略（Computer 不再调用）

**结论**：v0.2 Computer 对旧 MCP Server 宽容（降级运行），但 v0.1 Computer **无法**消费 v0.2 MCP Server。**升级顺序**：Computer 先于 MCP Server。

### 7.4 建议的升级顺序

```
阶段 1：发布 v0.2 Computer SDK（含 DPE Resolver API + get_dpe + get_resources）
阶段 2：用户升级 Computer 到 v0.2，业务方注册 Resolver
阶段 3：发布 v0.2 MCP Server Helper（Window 用 annotations，DPE 用 _meta）
阶段 4：MCP Server 增量升级（Window/DPE 元数据迁移）
阶段 5：发布 v0.2 Agent SDK（含 get_dpe / get_resources 客户端 + 应用层拉取）
阶段 6：Agent 升级；旧 v0.1 客户端无法连接 v0.2 Server，由 4008 提示用户升级
```
